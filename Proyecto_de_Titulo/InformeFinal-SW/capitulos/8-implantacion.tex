Este capítulo describe la implantación y puesta en marcha de la versión de prueba del sistema \textbf{DOMU}, alineando la estrategia de despliegue con la arquitectura y el desarrollo documentados en los capítulos anteriores. A diferencia de un escenario de producción sobre nube pública, la validación operativa del sistema se realizará sobre infraestructura institucional provista por la Universidad del Bío-Bío, lo que permite contar con un entorno controlado para pruebas técnicas, demostraciones funcionales y marcha blanca.

\section{Arquitectura de Despliegue}

La implantación de DOMU adopta un esquema \textbf{cliente-servidor} coherente con la arquitectura definida en el Capítulo~6 y con la implementación descrita en el Capítulo~7. El \textit{frontend} se publica como una SPA/PWA accesible desde navegador, mientras que el \textit{backend} Java expone la API REST y los canales WebSocket requeridos por la plataforma. La persistencia se resuelve mediante MySQL~8, y los servicios externos continúan siendo utilizados para correo electrónico, almacenamiento documental y generación de archivos PDF.

\subsection{Componentes Desplegados}

Los componentes considerados en la implantación son los siguientes:

\begin{itemize}
    \item \textbf{Frontend Web:} La interfaz desarrollada con React~19 y Vite~7 se compila como una SPA y se publica en el servidor institucional a través del puerto web asignado. Esta misma publicación soporta la experiencia PWA descrita en el Capítulo~7.

    \item \textbf{Backend:} El servidor de aplicaciones se ejecuta con Java~21 y Javalin~6, exponiendo los más de 120~endpoints REST y los canales WebSocket \texttt{/ws/chat} y \texttt{/ws/notifications}.

    \item \textbf{Base de Datos:} La persistencia del sistema utiliza MySQL~8 con la base de datos \texttt{crivas\_bd}, manteniendo consistencia con el diseño de persistencia y las migraciones versionadas mediante Flyway.

    \item \textbf{Repositorio de código:} El control de versiones y almacenamiento del código fuente se realiza en el GitLab institucional (\texttt{gitlab.face.ubiobio.cl}), sirviendo como respaldo del proyecto y origen de los artefactos desplegados.

    \item \textbf{Servicios externos:} La implantación sigue dependiendo de Box para almacenamiento documental, Jakarta Mail/SMTP para notificaciones por correo y OpenPDF para la generación de comprobantes y estados de cuenta.
\end{itemize}

\section{Procedimiento de Puesta en Marcha}

La puesta en marcha del entorno de prueba se estructura en una secuencia controlada de actividades técnicas, con el fin de asegurar trazabilidad y reducir errores de configuración.

\subsection{Preparación del Servidor}

Previo al despliegue, el servidor virtual institucional debe quedar configurado con las dependencias necesarias para ejecutar la aplicación:

\begin{itemize}
    \item Instalación de Java~21 para la ejecución del backend.
    \item Disponibilidad de MySQL~8 y acceso a la base de datos \texttt{crivas\_bd}.
    \item Configuración del puerto web institucional para publicar la versión de prueba.
    \item Habilitación de acceso remoto vía SSH sobre el puerto \texttt{1897}.
    \item Definición de variables de entorno para conexión a base de datos, JWT, correo electrónico, Box y configuración general del servidor.
\end{itemize}

Estas variables son coherentes con la configuración descrita en la clase \texttt{AppConfig} del \textit{backend}, evitando el uso de credenciales embebidas en el código fuente.

\subsection{Despliegue de Artefactos}

El proceso de implantación contempla la publicación separada de los dos artefactos principales del sistema:

\begin{enumerate}
    \item Construcción del \textit{frontend} con Vite para obtener los archivos estáticos optimizados.
    \item Compilación del \textit{backend} con Gradle y generación del artefacto ejecutable en Java.
    \item Transferencia de los artefactos al servidor virtual institucional.
    \item Publicación del \textit{frontend} en el puerto web asignado por la Universidad.
    \item Levantamiento del \textit{backend} y verificación de conectividad con la base de datos y los servicios externos.
\end{enumerate}

Una vez desplegados ambos componentes, se realizan pruebas de humo sobre autenticación, navegación general, conexión WebSocket y persistencia de datos.

\subsection{Gestión de Migraciones}

El esquema de base de datos se mantiene versionado mediante \textbf{Flyway}. Al iniciar el servidor, se ejecutan automáticamente las migraciones pendientes, garantizando que la estructura de \texttt{crivas\_bd} se encuentre sincronizada con la versión del código desplegado. Esta estrategia reduce el riesgo de inconsistencias entre la implementación del \textit{backend} y la persistencia real utilizada durante la marcha blanca.

\section{Plan de Capacitación}

La adopción del sistema requiere acompañar a los distintos perfiles de usuario con actividades breves de inducción, ajustadas a sus responsabilidades dentro de la comunidad.

\subsection{Administración y Comité}

Se contempla una sesión de inducción de aproximadamente 2~horas, orientada a la configuración inicial del edificio y a la operación de los módulos de gestión y gobernanza. Esta capacitación considera:

\begin{itemize}
    \item Carga de residentes y unidades habitacionales.
    \item Configuración de períodos de gasto común y consulta de estados de cuenta.
    \item Uso del módulo de votaciones.
    \item Publicación de documentos comunitarios, incluyendo actas.
    \item Validación de flujos de revisión por parte del Comité.
\end{itemize}

Se entrega además un manual digital de operación, segmentado por rol.

\subsection{Personal de Conserjería}

La capacitación para conserjería es principalmente práctica y de corta duración (aproximadamente 30~minutos). Se concentra en:

\begin{itemize}
    \item Inicio de sesión.
    \item Registro y verificación de visitas.
    \item Uso de lectura QR para control de acceso.
    \item Registro de encomiendas.
\end{itemize}

\subsection{Residentes}

Para residentes se considera una estrategia liviana de adopción, basada en material audiovisual breve y apoyo remoto. El contenido cubre registro, recuperación de contraseña, autorización de visitas, reservas de áreas comunes y consulta de gastos comunes.

\section{Estrategia de Lanzamiento}

La implantación se realizará mediante una \textbf{marcha blanca} de dos semanas sobre el entorno institucional de prueba, con el objetivo de validar tanto el funcionamiento técnico como la adopción gradual de los usuarios.

\begin{enumerate}
    \item \textbf{Semana~1:} Operación en paralelo con los mecanismos tradicionales. El objetivo es validar autenticación, acceso, encomiendas y carga inicial de datos.

    \item \textbf{Semana~2:} Transición progresiva hacia el uso habitual de la plataforma para visitas, encomiendas, reservas y notificaciones.

    \item \textbf{Cierre de marcha blanca:} Revisión de incidencias, ajustes menores de configuración y habilitación formal del entorno de prueba para demostraciones y validación académica.
\end{enumerate}

\section{Respaldo y Recuperación}

Con el fin de proteger la información generada durante la implantación de prueba, se define la siguiente estrategia de respaldo:

\begin{itemize}
    \item \textbf{Base de datos:} Generación periódica de respaldos de MySQL mediante \texttt{mysqldump}, almacenados en un directorio protegido del servidor y/o en un medio institucional secundario.

    \item \textbf{Archivos y documentos:} Los archivos cargados por la aplicación se almacenan en Box, aprovechando las capacidades de versionado y recuperación del servicio.

    \item \textbf{RPO:} Se establece como objetivo de punto de recuperación un máximo de 24~horas.

    \item \textbf{RTO:} Se establece como objetivo de tiempo de recuperación un máximo de 2~horas para restaurar la operación del entorno de prueba.
\end{itemize}

\section{Monitoreo y Mantenimiento}

El monitoreo del entorno de prueba se basa en mecanismos simples y coherentes con el contexto institucional del despliegue:

\begin{itemize}
    \item \textbf{Logs de aplicación:} Revisión de logs del backend para detectar errores HTTP, fallos de autenticación y eventos de WebSocket.

    \item \textbf{Verificación funcional:} Comprobación periódica de los flujos críticos del sistema, incluyendo inicio de sesión, acceso a módulos principales y consulta de datos persistidos.

    \item \textbf{Mantenimiento correctivo:} Los ajustes y correcciones se aplican mediante la generación de una nueva versión de los artefactos y su redepliegue controlado sobre el servidor institucional.

    \item \textbf{Mantenimiento evolutivo:} Las nuevas funcionalidades continúan desarrollándose en ramas independientes del repositorio institucional antes de su integración al entorno de prueba.
\end{itemize}

\section{Plan de Rollback}

En caso de detectarse fallos críticos durante la implantación, el procedimiento de reversión considera:

\begin{enumerate}
    \item Restaurar la última versión estable del \textit{frontend} y del \textit{backend}.
    \item Recuperar el respaldo más reciente de la base de datos si existiera incompatibilidad a nivel de esquema o datos.
    \item Verificar el funcionamiento de autenticación, persistencia y conectividad general.
    \item Registrar la incidencia y posponer la nueva publicación hasta corregir el error detectado.
\end{enumerate}

\section{Limitaciones del Despliegue}

La estrategia de implantación definida para la versión de prueba presenta las siguientes limitaciones:

\begin{itemize}
    \item \textbf{Entorno de prueba, no productivo:} La infraestructura institucional se utilizará para validación, demostración y marcha blanca, por lo que no corresponde a un escenario de alta disponibilidad ni de explotación comercial.

    \item \textbf{Despliegue manual:} La publicación de nuevas versiones se realiza manualmente a partir de artefactos generados por el equipo de desarrollo. La automatización mediante GitLab CI/CD queda propuesta como trabajo futuro.

    \item \textbf{Instancia única:} El servidor opera como una sola instancia, sin balanceo de carga ni redundancia horizontal.

    \item \textbf{Restricciones de la PWA:} La aplicación conserva las limitaciones propias de una Progressive Web App respecto de ciertas capacidades nativas del dispositivo.
\end{itemize}
