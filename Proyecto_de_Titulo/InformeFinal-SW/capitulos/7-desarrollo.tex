Este capítulo detalla el proceso de construcción del sistema \textbf{DOMU}, describiendo la estructura técnica del código fuente, la configuración del entorno, las fases de implementación ejecutadas bajo la metodología híbrida definida y los aspectos técnicos más relevantes de la solución. Se aborda la evolución del software desde su núcleo de autenticación hasta los módulos de gestión comunitaria, financiera y de comunicación en tiempo real.

\section{Estructura del Código y Arquitectura}
\label{sec:estructura_codigo}

El sistema se ha desarrollado siguiendo una arquitectura cliente-servidor desacoplada. El Backend expone una API REST con más de 120 endpoints, consumida por el Frontend construido como una \textit{Single Page Application} (SPA). Ambos repositorios se gestionan de forma independiente mediante Git, lo que permite ciclos de desarrollo, pruebas y despliegue autónomos.

\subsection{Backend (Java)}
El servidor se ha construido con \textbf{Java~21} y el framework ligero \textbf{Javalin~6}~\cite{javalin2025}, gestionado con \textbf{Gradle} (Kotlin DSL). El proyecto cuenta con aproximadamente 190~clases Java organizadas en una arquitectura por capas. La estructura principal bajo el paquete raíz \texttt{com.domu} es la siguiente:

\begin{itemize}
    \item \textbf{config/}: Configuración global del sistema. La clase \texttt{AppConfig} centraliza los parámetros del servidor, base de datos, JWT, correo electrónico y almacenamiento en Box. La clase \texttt{DependencyInjectionModule} configura el contenedor de inyección de dependencias mediante \textbf{Google Guice}, registrando como \textit{singletons} los 27~servicios, 21~repositorios, los manejadores WebSocket y el módulo de seguridad.

    \item \textbf{domain/}: Contiene 35~entidades de dominio organizadas en subpaquetes temáticos que reflejan los módulos del sistema:
    \begin{itemize}
        \item \texttt{core/}: Entidades base (\texttt{User}, \texttt{Building}, \texttt{HousingUnit}, \texttt{Role}).
        \item \texttt{access/}: Control de acceso (\texttt{Visit}, \texttt{VisitAuthorization}, \texttt{AccessLog}, \texttt{DeliveryPackage}).
        \item \texttt{finance/}: Gestión financiera (\texttt{CommonExpensePeriod}, \texttt{CommonCharge}, \texttt{CommonPayment}, \texttt{DelinquencyRecord}).
        \item \texttt{facility/}: Áreas comunes (\texttt{CommonArea}, \texttt{Reservation}).
        \item \texttt{voting/}: Votaciones (\texttt{VotingEvent}, \texttt{VotingOption}, \texttt{Vote}).
        \item \texttt{community/}: Foro (\texttt{ForumCategory}, \texttt{ForumThread}, \texttt{ForumMessage}).
        \item \texttt{staff/}: Personal (\texttt{Personnel}, \texttt{Shift}, \texttt{Task}, \texttt{PersonnelLog}).
        \item \texttt{ticket/}: Incidentes (\texttt{Ticket}, \texttt{TicketFollowUp}).
    \end{itemize}

    \item \textbf{dto/}: Aproximadamente 99~\textit{Data Transfer Objects} que definen los contratos de la API. Cada módulo cuenta con DTOs de solicitud (\texttt{*Request}) y respuesta (\texttt{*Response}), separando la representación HTTP de las entidades de dominio. Ejemplos representativos incluyen \texttt{LoginRequest}, \texttt{AuthResponse}, \texttt{CreateVisitRequest}, \texttt{CommonExpensePeriodDetailResponse} y \texttt{ChatMessageResponse}.

    \item \textbf{database/}: Capa de persistencia compuesta por 21~repositorios que implementan el acceso a datos mediante JDBC directo sobre MySQL, utilizando \textbf{HikariCP}~5.1 para el \textit{pool} de conexiones. Las clases principales incluyen \texttt{UserRepository}, \texttt{CommonExpenseRepository}, \texttt{VisitRepository}, \texttt{PollRepository}, \texttt{AmenityRepository}, \texttt{ChatRepository}, \texttt{ParcelRepository}, \texttt{NotificationRepository}, entre otros. La clase \texttt{DataSourceFactory} centraliza la creación del \textit{pool} de conexiones.

    \item \textbf{service/}: Capa de lógica de negocio con 27~servicios que encapsulan las reglas del sistema. Los servicios más relevantes son:
    \begin{itemize}
        \item \texttt{UserService}: Registro, autenticación, confirmación por correo y recuperación de contraseñas.
        \item \texttt{BuildingService}: Flujo de registro de comunidades con aprobación por correo e invitación al administrador.
        \item \texttt{CommonExpenseService}: Creación de períodos de gasto común, cargos por unidad y procesamiento de pagos.
        \item \texttt{CommonExpensePdfService}, \texttt{PaymentReceiptPdfService}, \texttt{ChargeReceiptPdfService}: Generación de documentos PDF con \textbf{OpenPDF}.
        \item \texttt{VisitService}: Autorización de visitas, generación de códigos QR y \textit{check-in}.
        \item \texttt{ChatService}: Gestión de salas de chat y mensajes en tiempo real.
        \item \texttt{NotificationService}: Creación y distribución de notificaciones vía WebSocket.
    \end{itemize}

    \item \textbf{security/}: Módulo de seguridad compuesto por 4~clases: \texttt{JwtProvider} (generación y validación de tokens JWT con HMAC256), \texttt{BCryptPasswordHasher} (hashing de contraseñas), \texttt{PasswordHasher} (interfaz de abstracción) y \texttt{AuthenticationHandler} (\textit{middleware} de Javalin que intercepta las rutas protegidas y valida el token en la cabecera \texttt{Authorization}).

    \item \textbf{email/}: Servicio de correo electrónico con 6~clases: la interfaz \texttt{EmailService}, su implementación SMTP (\texttt{SmtpEmailService}) y cuatro plantillas HTML para confirmación de cuenta, aprobación de comunidades, notificación de aprobación y notificación de rechazo.

    \item \textbf{web/}: Capa de presentación HTTP con 4~clases. \texttt{WebServer} es la clase principal que registra los más de 120~endpoints REST y los dos canales WebSocket (\texttt{/ws/chat} y \texttt{/ws/notifications}). \texttt{ChatWebSocketHandler} y \texttt{NotificationWebSocketHandler} gestionan la comunicación bidireccional en tiempo real. \texttt{UserMapper} convierte entidades de usuario a DTOs de respuesta.
\end{itemize}

Para mantener consistencia con el alcance funcional vigente del producto defendido, esta sección describe únicamente los módulos activos de autenticación, operación comunitaria, finanzas y gobernanza del Comité. Artefactos históricos fuera de dicho alcance no se consideran parte de la entrega final documentada.

El esquema de base de datos se versiona mediante \textbf{Flyway}, con 35~migraciones SQL que documentan la evolución del modelo de datos desde la creación del esquema de autenticación (V001) hasta las últimas actualizaciones del sistema de notificaciones y estructura de edificios (V035).

\subsection{Frontend (React)}
La interfaz de usuario se ha construido con \textbf{React~19} y el empaquetador \textbf{Vite~7}, utilizando JavaScript (JSX) y hojas de estilo SCSS. El proyecto comprende aproximadamente 92~archivos fuente con más de 20.000~líneas de código, organizados de la siguiente manera:

\begin{itemize}
    \item \textbf{pages/} (54~vistas): Componentes de nivel de ruta que conforman las pantallas de la aplicación. Están segmentadas por rol:
    \begin{itemize}
        \item \textit{Públicas} (7): \texttt{Home}, \texttt{Login}, \texttt{Register}, \texttt{ForgotPassword}, \texttt{About}, \texttt{Contact} y páginas de solución por tipo de usuario.
        \item \textit{Administrador} (13): \texttt{Dashboard}, \texttt{AdminCreateUser}, \texttt{AdminIncidentsBoard}, \texttt{AdminIncidentStats}, \texttt{AdminResidents}, \texttt{AdminHousingUnits}, \texttt{AdminAmenities}, \texttt{AdminCommonExpenses}, \texttt{AdminParcels}, \texttt{AdminTasks}, \texttt{AdminStaff} y vistas de gobernanza comunitaria reutilizadas por el Comité.
        \item \textit{Residente} (18): \texttt{ResidentProfile}, \texttt{ResidentVisits}, \texttt{ResidentIncidents}, \texttt{ResidentAmenities}, \texttt{ResidentCartola}, \texttt{ResidentChargesDetail}, \texttt{ResidentPaymentFlow}, \texttt{ResidentParcels}, \texttt{ResidentPublications}, \texttt{ResidentLibrary}, \texttt{ResidentMarketplace}, \texttt{ChatHub}, \texttt{Votaciones}, entre otras.
        \item \textit{Comité}: Acceso a capacidades de gobernanza comunitaria (votaciones, actas y consulta de estados de cuenta) mediante permisos sobre módulos existentes.
        \item \textit{Especiales}: Páginas de soluciones por tipo de usuario, confirmación de correo, registro de administrador por invitación y centro de notificaciones.
    \end{itemize}

    \item \textbf{components/} (18~componentes): Elementos reutilizables de la interfaz, incluyendo \texttt{Button} (con estados de carga), \texttt{FormField} (con validación), \texttt{Spinner}, \texttt{Skeleton} (\textit{skeleton loading}), \texttt{ConfirmModal}, \texttt{LocationPicker} (selector de ubicación basado en mapas mediante la librería \texttt{pigeon-maps}), \texttt{VisitRegistrationPanel}, \texttt{MarketCard}, \texttt{ProtectedRoute} y un módulo completo de pago (\texttt{PaymentMethodModal}, \texttt{CardPaymentView}, \texttt{BankTransferView}, \texttt{ChargeSelector}, \texttt{PaymentSuccess}).

    \item \textbf{layout/} (7~componentes): Estructura visual de la aplicación, compuesta por \texttt{ProtectedLayout} (envolvente para rutas autenticadas con barra lateral y cabecera), \texttt{AuthLayout} (envolvente para páginas públicas), \texttt{Header}, \texttt{Sidebar} (navegación filtrada por rol), \texttt{Footer} y \texttt{MainContent}.

    \item \textbf{services/}: Capa de comunicación con el Backend. El archivo principal \texttt{api.js} (1.034~líneas) centraliza todas las llamadas HTTP organizadas por módulo (\texttt{auth}, \texttt{visits}, \texttt{incidents}, \texttt{finance}, \texttt{polls}, \texttt{amenities}, \texttt{chat}, \texttt{market}, \texttt{parcels}, \texttt{library}, \texttt{notifications}, entre otros). Cada petición inyecta automáticamente el token JWT en la cabecera \texttt{Authorization} y el identificador de la comunidad activa en la cabecera \texttt{X-Building-Id}.

    \item \textbf{context/}: Estado global de la aplicación mediante \textbf{React Context API}. El \texttt{AppContext} gestiona la sesión del usuario autenticado, el edificio seleccionado (soporte multi-comunidad), el tema visual (claro/oscuro) y funciones de \textit{logout} y cambio de comunidad.

    \item \textbf{hooks/}: \textit{Custom hooks} de React. El hook \texttt{useNotifications} integra WebSocket con \textbf{TanStack React Query}~v5 para recibir notificaciones en tiempo real, con reconexión automática cada 5~segundos ante desconexiones.

    \item \textbf{constants/}: Configuración centralizada con 85~definiciones de rutas, secciones de navegación filtradas por rol y 20~tipos de notificación con iconos y colores asociados.

    \item \textbf{styles/}: Hojas de estilo globales en SCSS compiladas por Vite.
\end{itemize}

La aplicación se configura como \textit{Progressive Web App} (PWA) mediante el plugin \texttt{vite-plugin-pwa}, y utiliza TanStack React Query para la gestión de caché y revalidación de datos del servidor.


\section{Iteraciones en el desarrollo}
\label{sec:iteraciones_desarrollo}

Siguiendo el enfoque incremental definido para el proyecto, el desarrollo del sistema DOMU se organizó en iteraciones sucesivas. Cada iteración incorporó nuevos módulos funcionales sobre la infraestructura previamente construida, permitiendo validar progresivamente el funcionamiento del sistema.

Este enfoque permitió reducir la complejidad técnica del desarrollo y facilitar la integración gradual de los distintos componentes del sistema. A continuación se describen las seis iteraciones principales que estructuraron el proceso de implementación.

\subsection{Primera Iteración: Infraestructura Base y Seguridad}

La primera iteración se centró en establecer la infraestructura técnica fundamental del sistema.

Durante esta etapa se implementaron los siguientes componentes:

\begin{itemize}

\item Configuración inicial del proyecto backend utilizando Java 21, Gradle y el framework Javalin.

\item Implementación de la conexión a la base de datos MySQL mediante HikariCP.

\item Integración del sistema de migraciones de base de datos utilizando Flyway, permitiendo versionar el esquema y asegurar su consistencia entre entornos.

\item Configuración del contenedor de inyección de dependencias mediante Google Guice.

\item Implementación del sistema de autenticación basado en JSON Web Tokens (JWT), incluyendo registro de usuarios, inicio de sesión y validación de sesiones.

\item Implementación del flujo de confirmación de cuentas por correo electrónico y recuperación segura de contraseñas.

\end{itemize}

Esta iteración permitió establecer la base técnica necesaria para el desarrollo posterior de los módulos funcionales del sistema.

\subsection{Segunda Iteración: Gestión de Comunidades y Usuarios}

En la segunda iteración se desarrollaron los componentes necesarios para la administración básica de comunidades residenciales.

Las principales funcionalidades implementadas fueron:

\begin{itemize}

\item Registro de comunidades mediante un flujo de solicitud y aprobación.

\item Integración con el servicio de almacenamiento en la nube Box para guardar documentación asociada al registro de comunidades.

\item Implementación del soporte multi-comunidad, permitiendo que un usuario pueda pertenecer a múltiples edificios.

\item Desarrollo del modelo de unidades habitacionales y su relación con los residentes.

\item Implementación del sistema de roles del sistema: Administrador, Residente, Conserje, Personal y Comité.

\item Desarrollo de las vistas segmentadas por rol dentro de la interfaz de usuario.

\end{itemize}

Esta iteración permitió configurar completamente la estructura organizacional de las comunidades dentro del sistema.

\subsection{Tercera Iteración: Operación Comunitaria}

La tercera iteración incorporó los módulos orientados a la operación diaria de la comunidad.

Entre las funcionalidades implementadas se encuentran:

\begin{itemize}

\item Registro y gestión de incidentes reportados por residentes.

\item Sistema de reservas de áreas comunes con control de disponibilidad y reglas de uso.

\item Implementación del módulo de votaciones comunitarias.

\item Desarrollo del sistema de control de accesos mediante registro de visitas.

\item Integración del lector de código QR de la cédula de identidad para facilitar el registro automático de visitantes.

\end{itemize}

Estas funcionalidades constituyen los procesos operativos centrales de la gestión comunitaria.

\subsection{Cuarta Iteración: Módulo Financiero}

En la cuarta iteración se implementó el sistema de gestión financiera asociado a los gastos comunes.

Las funcionalidades desarrolladas incluyen:

\begin{itemize}

\item Creación de períodos de gastos comunes.

\item Generación de cargos asociados a cada unidad habitacional.

\item Consulta de estados de cuenta por parte de los residentes.

\item Registro de pagos mediante carga de comprobantes de transferencia.

\item Generación automática de documentos PDF como estados de cuenta y comprobantes de pago.

\end{itemize}

Este módulo representó uno de los componentes más complejos del sistema debido a la lógica de negocio involucrada en el cálculo y administración de los cargos.

\subsection{Quinta Iteración: Comunicación y Vida Comunitaria}

La quinta iteración se enfocó en los módulos de interacción social dentro de la comunidad.

Durante esta etapa se implementaron los siguientes componentes:

\begin{itemize}

\item Sistema de chat entre residentes utilizando WebSockets para comunicación en tiempo real.

\item Muro comunitario para la publicación de avisos y comunicaciones internas.

\item Marketplace comunitario para la publicación de productos entre vecinos.

\item Sistema de gestión de encomiendas para registrar la recepción y entrega de paquetes.

\item Biblioteca digital para almacenar documentos relevantes de la comunidad.

\end{itemize}

Estas funcionalidades fortalecen la comunicación y la interacción entre los miembros de la comunidad.

\subsection{Sexta Iteración: Gestión Operativa y Notificaciones}

La última iteración se centró en completar los módulos operativos del sistema y consolidar la arquitectura de notificaciones.

Las principales funcionalidades incorporadas fueron:

\begin{itemize}

\item Sistema de gestión de personal de la comunidad.

\item Asignación y seguimiento de tareas para funcionarios.

\item Implementación de notificaciones en tiempo real mediante WebSockets.

\item Desarrollo del centro de notificaciones dentro de la aplicación.

\item Incorporación de funcionalidades de gobernanza del Comité, incluyendo votaciones y consulta de estados de cuenta.

\item Integración de la documentación interactiva de la API mediante Swagger UI.

\end{itemize}

Con esta iteración el sistema alcanzó un estado funcional completo, integrando los distintos módulos del sistema en una plataforma coherente para la administración de comunidades residenciales.

\section{Riesgos y Problemas}

Durante el desarrollo del sistema DOMU se identificaron diversos riesgos de alto nivel que podían afectar el avance del proyecto, la calidad del software o la estabilidad del sistema. 
La identificación temprana de estos riesgos permitió definir estrategias de mitigación orientadas a reducir su probabilidad de ocurrencia o minimizar su impacto en caso de materializarse.

A continuación se describen los principales riesgos identificados durante el desarrollo del proyecto, junto con su impacto potencial y la estrategia adoptada para su mitigación.

\subsection*{Complejidad del modelo de datos}

El sistema DOMU integra múltiples módulos funcionales —como control de accesos, gestión financiera, votaciones, incidentes y comunicación comunitaria— que comparten información dentro de una misma base de datos. 
La evolución simultánea de estos módulos representaba un riesgo de inconsistencias en el modelo de datos o pérdida de integridad referencial.

\textbf{Impacto:} posibles errores en la persistencia de datos, inconsistencias en registros financieros o problemas en la relación entre entidades del sistema.

\textbf{Estrategia de mitigación:} se implementó un sistema de migraciones versionadas mediante la herramienta Flyway, permitiendo mantener trazabilidad de los cambios en el esquema de base de datos y asegurar la consistencia del modelo a lo largo del desarrollo.

\subsection*{Integración con hardware de lectura QR}

El sistema incorpora un lector de códigos QR utilizado en conserjería para registrar visitas mediante la cédula de identidad chilena. 
La integración con este hardware presentaba incertidumbre respecto al formato de datos entregado por el dispositivo y su compatibilidad con navegadores web.

\textbf{Impacto:} fallas en el registro automático de visitantes o necesidad de realizar procesos manuales de registro.

\textbf{Estrategia de mitigación:} se utilizó el modo de emulación de teclado del lector, permitiendo capturar directamente la cadena de texto del código QR desde el navegador y procesarla mediante lógica de parsing en el frontend antes de enviarla al backend.

\subsection*{Desincronización entre frontend y backend}

Debido a que el frontend y el backend se desarrollan en repositorios independientes, existía el riesgo de inconsistencias entre los contratos de la API y su consumo desde la interfaz de usuario.

\textbf{Impacto:} errores en llamadas HTTP, fallas de visualización en la interfaz o incompatibilidad entre versiones de los módulos.

\textbf{Estrategia de mitigación:} se adoptó documentación interactiva de la API mediante Swagger UI, permitiendo validar endpoints durante el desarrollo y facilitar la integración entre ambos componentes del sistema.

\subsection*{Complejidad del módulo financiero}

La gestión de gastos comunes incorpora reglas de negocio complejas, como la generación de cargos por unidad habitacional, registro de pagos y generación de documentos financieros.

\textbf{Impacto:} errores en cálculos financieros o inconsistencias en estados de cuenta de los residentes.

\textbf{Estrategia de mitigación:} se implementaron servicios especializados para el cálculo de cargos y generación de documentos PDF, además de realizar pruebas funcionales sobre los flujos de generación de estados de cuenta.

\subsection*{Escalabilidad y rendimiento del sistema}

El sistema debe soportar múltiples usuarios accediendo simultáneamente a funcionalidades como chat, notificaciones o consultas de datos.

\textbf{Impacto:} degradación del rendimiento del sistema o lentitud en la experiencia de usuario.

\textbf{Estrategia de mitigación:} se implementó una arquitectura basada en API REST desacoplada y el uso de WebSockets únicamente para funcionalidades que requieren comunicación en tiempo real, reduciendo la carga sobre el servidor.

Estos componentes requirieron un mayor esfuerzo de diseño y pruebas para asegurar su correcto funcionamiento.

En términos generales, el proyecto permitió consolidar conocimientos en arquitectura de software, desarrollo de APIs, integración de servicios externos y diseño de interfaces web modernas.
